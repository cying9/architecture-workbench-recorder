# architecture-workbench-recorder v0.1 Design

## 1. Summary

`architecture-workbench-recorder` is a local-first CLI tool that converts a thread or transcript into a structured workbench update proposal, then optionally applies approved updates back into an existing Markdown architecture workbench.

It is explicitly:

- conversation-driven
- local-first
- markdown-first
- proposal-first
- human-in-the-loop

It is explicitly not:

- a generic meeting notes product
- an automatic architecture decision maker
- a graphify replacement
- a review runtime rewrite

The v0.1 job is narrow and practical:

1. ingest a transcript
2. compare it to an existing workbench
3. produce a human-readable proposal bundle
4. optionally write approved changes into the target Markdown files with minimal intrusion

## 2. Product Goals

### Primary Goal

Help a human maintain an architecture workbench as a stable thinking scaffold while a long-running design thread continues to evolve.

### Secondary Goals

- reduce post-hoc manual consolidation work
- preserve evolving terminology with traceability
- surface conflicts and duplicates before writing
- keep the workbench lightweight rather than over-systematized

## 3. Non-Goals

V0.1 does not aim to provide:

- speech-to-text
- calendar, meeting, or task management
- strong coupling to any specific review runtime
- autonomous architectural judgment
- mandatory LangGraph orchestration
- multi-agent conversation management
- enterprise collaboration workflows

## 4. Design Principles

- `local-first`: transcripts, proposals, and apply operations run locally by default
- `markdown-first`: the workbench remains ordinary Markdown files, readable and maintainable without the tool
- `proposal-first`: the default action is proposal generation, not document mutation
- `structure-preserving`: update existing workbench structure instead of flattening it into a new format
- `human-in-the-loop`: the tool proposes; the human accepts or rejects
- `lightweight workbench`: the workbench stays small, selective, and design-oriented
- `conversation-to-workbench`: transcript material is transformed into durable workbench state
- `minimal intrusion into existing docs`: prefer append and targeted edits over wholesale rewrites

## 5. Observed Current Workbench Shape

The current workbench sample is a five-file system with distinct roles.

### `README.md`

Acts as the workbench front door. It carries:

- current status and phase
- current system map
- top-level principles
- current architecture framing
- links to the other files

It is curated and high-signal. It should not be rewritten frequently.

### `glossary.md`

Stores relatively stable working definitions. It prefers stable meanings over full coverage. Entries are term-centric, not thread-centric.

### `decision-log.md`

Stores short architecture decisions and working judgments. Entries have IDs, date grouping, status, decision, and rationale. It is append-oriented and should separate stable judgments from unresolved discussion.

### `open-questions.md`

Stores unresolved design questions only. It is a living set of active questions, not a speculative brainstorm dump.

### `thread-log.md`

Stores selective milestones from the thread rather than verbatim notes. It is closer to architectural storyline than summary.

### Implication For The Tool

The recorder should not generate one giant summary. It must propose updates into distinct semantic slots:

- stable term definitions
- decisions or working decisions
- unresolved questions
- thread milestones
- selective README suggestions only when the top-level framing has clearly shifted

## 6. Reuse Analysis

### `graphify`

Useful ideas to borrow:

- persistent artifact mindset
- incremental `--update` orientation
- explicit audit trail
- transcript as source material

Do not borrow:

- graph as the primary product
- community detection or graph-centric UX as the main abstraction

Conclusion:
`graphify` is a useful reference for staged pipelines, persistence, and honesty, but not for the product center of gravity.

### `doc-coauthoring`

Useful ideas to borrow:

- structured human-in-the-loop workflow
- section-aware document collaboration
- emphasis on refinement rather than one-shot generation

Do not borrow:

- interactive document drafting as the main task model
- freeform co-authoring loop as the core runtime shape

Conclusion:
Useful for proposal presentation and apply discipline, but not as the engine model.

### `intake`

Useful ideas to borrow:

- capability-directory layout
- clean separation of `config`, `source_code`, `runtime`, and docs
- CLI-first module organization

Conclusion:
Strong template for repository structure.

### `doc_parse`

Useful ideas to borrow:

- normalized intermediate artifact before downstream use
- Markdown-oriented bundle output

Conclusion:
Important for proposal bundle design: normalize first, then compare and render.

## 7. Candidate Solution Approaches

### Option A: Deterministic Rule-First Recorder

Pipeline:

1. parse transcript
2. use heuristics and regex-like rules to extract candidate items
3. compare with Markdown files
4. generate Markdown proposal

Strengths:

- simple
- transparent
- low dependency surface
- predictable behavior

Weaknesses:

- brittle for evolving terminology
- weak at long-range thread interpretation
- high manual tuning cost
- likely to underperform on nuanced `working_hypothesis` vs `decided` distinctions

Best fit:

- very narrow domains with rigid transcript conventions

### Option B: Explicit Stage Pipeline With Provider-Agnostic LLM Extraction

Pipeline:

1. ingest transcript
2. normalize turns
3. extract candidate items with explicit schemas
4. classify and score confidence
5. compare against existing workbench snapshot
6. render proposal bundle
7. apply selected changes with deterministic Markdown operations

Strengths:

- best balance of flexibility and control
- preserves explicit state and stage contracts
- provider-agnostic at the extraction layer
- deterministic compare/apply path
- well aligned with `proposal-first`

Weaknesses:

- more moving parts than pure deterministic extraction
- requires careful schema discipline
- extraction quality depends on prompts and provider behavior

Best fit:

- the stated v0.1 requirements

### Option C: Graph-Native Recorder

Pipeline:

1. ingest transcript
2. build thread graph of claims, terms, questions, and milestones
3. compare graph against workbench graph state
4. derive Markdown deltas

Strengths:

- rich long-term extensibility
- better foundation for future cross-thread memory
- potentially strong for conflict tracing and term drift

Weaknesses:

- too heavy for v0.1
- easy to drift toward graphify duplication
- graph maintenance becomes product work before proposal/apply is proven

Best fit:

- a later v0.3+ evolution if the workbench recorder proves valuable

## 8. Recommended Approach

Recommend **Option B: explicit stage pipeline with provider-agnostic LLM extraction plus deterministic compare/apply**.

Why this is the best fit:

- it matches the product center: transcript to workbench proposal
- it keeps the system small and legible
- it preserves explicit contracts without over-agentizing the implementation
- it leaves room for future graph-shaped internals without making graph state the product today

In short:

- extraction can be LLM-assisted
- state and schemas should be explicit
- comparison and apply should be deterministic
- human-readable proposal remains the main output

## 11. Input Model

### Transcript Model

Normalized turn schema:

```text
TranscriptTurn
- turn_id
- speaker
- role
- timestamp?
- raw_text
- normalized_text
- source_span
```

The tool should accept loose transcript input, then normalize it into an internal turn list before extraction.

### Workbench Snapshot Model

```text
WorkbenchSnapshot
- root_path
- readme
- glossary_entries[]
- decisions[]
- open_questions[]
- thread_entries[]
```

This snapshot is parsed from the existing Markdown files. V0.1 should parse only the target files it owns.

## 12. Candidate Item Model

```text
CandidateItem
- candidate_id
- type
- title
- normalized_key
- statement
- rationale?
- evidence_turn_ids[]
- confidence
- status_hint?
- proposed_target_file
- conflict_flags[]
```

Design note:

- `normalized_key` is crucial for duplicate detection
- `evidence_turn_ids` provides auditability back to the transcript
- `status_hint` helps distinguish `decided` vs `working_hypothesis`

## 13. Proposal Model

### Human-Readable Proposal

Main artifact:

- `proposal.md`

Recommended sections:

1. proposal summary
2. input sources
3. workbench snapshot summary
4. proposed glossary updates
5. proposed decision-log updates
6. proposed open-question updates
7. proposed thread-log updates
8. README suggestions
9. conflicts and duplicates
10. apply preview

### Machine-Readable Proposal

Optional artifact:

- `proposal.json`

Suggested top-level shape:

```text
ProposalBundle
- proposal_id
- created_at
- transcript_source
- workbench_root
- summary
- additions
- updates
- reopenings
- duplicates
- conflicts
- apply_plan
```

### Apply Unit Model

Every proposed write should be an explicit unit:

```text
ApplyUnit
- unit_id
- target_file
- operation
- anchor
- content
- rationale
- evidence_turn_ids[]
```

This keeps apply deterministic and reviewable.

## 14. Conflict Handling

V0.1 should explicitly detect and report at least these conflict classes.

### Term Definition Conflict

Example:

- transcript proposes a new definition for `object`
- glossary already defines `object` differently

Action:

- do not silently overwrite
- emit conflict
- propose either update or manual review

### Decision Conflict

Example:

- transcript implies the system now rejects a previously decided architecture stance

Action:

- mark as decision conflict
- require explicit human confirmation
- preserve prior decision entry unless the user approves a superseding entry

### Reopened Question

Example:

- a previously closed or resolved issue reappears as active uncertainty

Action:

- emit `reopened_question`
- prefer adding a note or new question entry over deleting history

### Duplicate Or Near-Duplicate

Example:

- new milestone repeats an already logged thread milestone
- new glossary candidate is a synonym of an existing term

Action:

- classify as duplicate or merge candidate
- avoid redundant append unless user chooses otherwise

### Ambiguous Stability

Example:

- transcript suggests something may be a decision, but evidence is weak

Action:

- downgrade to `working_hypothesis`
- keep confidence visible in proposal

## 15. Workbench File Update Policy

### `README.md`

Policy:

- conservative
- suggest changes only for top-level framing shifts
- default to proposal-only for README in v0.1

Examples of acceptable README suggestions:

- current phase changed materially
- top-level architecture framing changed
- core principles materially shifted

### `glossary.md`

Policy:

- append new stable terms
- propose edits for conflicting terms
- avoid flooding with low-stability jargon

### `decision-log.md`

Policy:

- append new entries
- allow `working decision` as explicit status
- never silently rewrite old decisions

### `open-questions.md`

Policy:

- add genuinely unresolved questions
- mark duplicates or reopened questions
- avoid converting weak discussion fragments into formal questions

### `thread-log.md`

Policy:

- append selective milestones
- prefer architectural shifts over ordinary back-and-forth
- treat this as storyline, not summary

## 16. CLI Design

Recommended CLI style:

```powershell
python source_code/awr/cli.py --help
```

Potential command surface for v0.1:

### `propose`

```powershell
python source_code/awr/cli.py propose --transcript <path> --workbench <dir>
```

Responsibilities:

- normalize transcript
- parse workbench
- extract candidates
- compare against current state
- emit proposal bundle

### `apply`

```powershell
python source_code/awr/cli.py apply --proposal <proposal-json-or-dir> --workbench <dir>
```

Responsibilities:

- validate proposal
- show apply summary
- write targeted file updates only
- create backups or write journal before mutation

### Optional Supporting Commands

These are useful but not strictly mandatory for day one:

- `inspect-workbench`
- `normalize-transcript`
- `validate-proposal`

## 17. Provider Strategy

V0.1 should be provider-agnostic.

The extraction layer should depend on an internal interface rather than a hard-coded provider.

Example conceptual contract:

```text
ExtractorBackend
- extract_candidates(normalized_transcript, workbench_snapshot, config) -> CandidateItem[]
```

This lets v0.1 begin with one backend while keeping the repository architecture open.

## 18. Apply Strategy

Apply must be intentionally conservative.

### Rules

1. default command path is `propose`
2. `apply` requires an explicit proposal artifact
3. do not rewrite the entire workbench
4. operate through per-file targeted edits
5. create pre-apply backup copies or journal entries
6. write changes in stable Markdown patterns
7. validate the resulting file set after mutation

### Recommended Apply Method

- parse existing target file into coarse structural blocks
- find insertion or update anchor
- write only the relevant block
- preserve surrounding text

For v0.1, append-plus-anchor is preferable to aggressive AST-like Markdown rewriting.

## 19. Minimal Test Strategy

### Unit Tests

- transcript normalization
- candidate classification
- duplicate detection
- conflict detection
- Markdown proposal rendering
- targeted apply operations

### Golden Tests

Use one or two stable sample transcript/workbench pairs and assert:

- proposal Markdown stays readable
- proposal JSON remains schema-valid
- apply produces expected target-file deltas

### Safety Tests

- conflicting glossary definition does not silently overwrite
- conflicting decision does not silently replace prior entry
- README is not globally rewritten

### Demo Test

A minimal end-to-end demo should prove:

1. transcript in
2. proposal bundle out
3. user-approved apply
4. workbench updated safely

## 21. Risks And Design Tensions

### Over-Summarization Risk

If the tool behaves like a meeting summarizer, it will miss the point of the workbench.

Mitigation:

- force extraction into explicit workbench categories
- keep thread-log selective

### Over-Automation Risk

If the tool mutates docs too eagerly, trust will collapse.

Mitigation:

- proposal-first default
- deterministic apply
- explicit conflict reporting

### Over-Structuring Risk

If the tool invents too much schema, the workbench will become harder to maintain than manual notes.

Mitigation:

- keep file model narrow
- favor append-oriented Markdown patterns

### Graph Drift Risk

If the design leans too hard into graph-native ideas too early, the repository will duplicate `graphify`.

Mitigation:

- keep graph-shaped thinking at the contract level only
- do not make graph persistence a v0.1 requirement

## 22. Final Recommendation

Build v0.1 as a **small, explicit, CLI-first recorder**:

- Python
- `typer` for CLI
- `pydantic` for contracts
- provider-agnostic extraction
- deterministic compare and apply
- Markdown proposal as the primary user-facing artifact

The repository should prove one thing well:

`transcript + existing workbench -> structured proposal -> safe apply`

That is enough for v0.1.
